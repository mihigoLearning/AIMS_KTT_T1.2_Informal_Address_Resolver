## Problem

Motorcycle delivery riders in Kigali receive addresses like *"inyuma ya big pharmacy on RN3, red gate"* in a mix of English, French, and Kinyarwanda. This resolver converts such free-text descriptions into `(lat, lon, confidence)` tuples — no LLM calls, CPU-only, sub-100 ms latency.

## How It Works

1. **Language detection** — heuristic word-set matching for EN/FR/Kinyarwanda
2. **Modifier extraction** — maps spatial words (`inyuma ya`, `derrière`, `next to`…) to bearing + distance
3. **Fuzzy landmark matching** — `rapidfuzz.partial_ratio` against 50-landmark gazetteer (names + FR/KIN aliases), cutoff = 68
4. **Coordinate offset** — spherical offset in modifier direction; confidence = fuzzy score × modifier bonus
5. **Escalation** — confidence < 0.45 → flagged for dispatcher

## Trade-offs

- `partial_ratio` over `token_sort_ratio`: better handles noisy prefix/suffix descriptions but more prone to false positives on very short inputs → mitigated by raising cutoff to 68.
- Simple bearing table over ML direction inference: explainable, fast, debuggable — and sufficient for Tier 1.
- No LLM calls: adds 0 latency, 0 API cost, works fully offline.

## Known Limitations

- `near` / `hafi ya` do not encode direction → defaults to bearing 0° (North); wrong ~75% of the time when used alone.
- Two-landmark descriptions (`"between A and B"`) resolve only to the higher-scoring single match.
- Emoji prefix can degrade fuzzy match score slightly.

## Dataset

Generated by `generate_data.py` (seed=42, reproducible). Files: `gazetteer.json` (50 landmarks), `descriptions.csv` (200 rows), `gold.csv` (50 rows).

## Model

No trained model — resolver uses fuzzy string matching only (rapidfuzz). No HuggingFace checkpoint.
